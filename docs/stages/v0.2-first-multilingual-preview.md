# InputFlow v0.2 First Multilingual Preview Spec

## Stage Goal

Make InputFlow useful beyond the original Dutch/Korean seed workflow while preserving the small, reliable tray-app model.

The v0.2 release should let a technical multilingual Windows user create a working config from installed input profiles without hand-authoring the whole JSON file.

## Positioning

InputFlow is a Windows input-profile workflow utility. It is not a translator, keyboard layout editor, macro suite, or broad IME automation tool.

The product promise for v0.2 is:

- choose exact installed Windows input profiles
- bind them to clear workflow behavior
- avoid blind cycling through Windows input methods
- enter Korean Hangul safely when configured
- explain profile matching problems through diagnostics

## In Scope

1. First-run setup path for creating an initial config.
2. Installed-profile picker based on the same inventory shown in diagnostics.
3. Workflow creation for:
   - two-profile toggle
   - N-profile cycle
   - direct switch
   - previous profile, if implementation stays small and testable
4. Trigger picker with explicit warnings for single-key triggers, especially RightAlt/AltGr.
5. Profile health states:
   - matched
   - missing
   - ambiguous
   - changed since config was written
6. Config write path that validates before saving.
7. Manual JSON remains supported.
8. README and validation docs updated to remove reliance on Dutch/Korean as the default product story.
9. Existing v0.1 behavior remains working, including RightAlt and Korean Hangul/native entry.

## Out Of Scope

1. Full settings editor for every config field.
2. Installer or auto-update.
3. Japanese or Chinese mode automation.
4. Per-app remembered profile behavior.
5. Cloud sync, accounts, telemetry, ads, or monetization hooks.
6. Code signing.
7. Heavy UI framework migration unless a specific constraint forces it.

## User Flows

### First Run

When no config exists, InputFlow creates an inventory-backed starter config and opens `Setup Status` instead of silently assuming the Dutch/Korean seed workflow.

Current v0.2 flow:

1. Generate configured profile entries from installed Windows input profiles.
2. Show installed input profiles, configured profile health, workflow readiness, and excluded processes.
3. Let the user add or remap configured profiles.
4. Let the user choose a workflow type.
5. Let the user choose profiles for that workflow when the mode requires targets.
6. Let the user choose a trigger.
7. Validate the generated config before saving.
8. Reload and register the saved workflow without restarting.

A complete standalone settings editor belongs to v0.3.

### Existing Config

When a config already exists, InputFlow continues using it. The user can open setup/profile selection from the tray `Setup Status` item.

### Profile Problems

If configured profiles are missing, ambiguous, or changed, InputFlow should not silently choose a surprising replacement. It should preserve the previous working runtime state where possible and surface the problem through diagnostics/logs.

## Data Model Direction

The current config schema should remain compatible for v0.2.

Additions should be conservative:

- Store enough profile identity to detect changed profiles.
- Prefer exact identifiers from Windows inventory over loose display names.
- Do not remove KLID/HKL compatibility unless replaced by a better tested identity.
- Any new fields must be ignored safely by older logic where practical.

## Implemented Slices

The following v0.2 slices are implemented on `main`:

- Inventory-backed starter config is created on first run.
- Setup Status opens automatically for new starter configs.
- Setup Status shows installed profiles, configured profile health, workflow readiness, and excluded processes.
- Profile picker can add profiles and remap existing profile IDs to exact installed Windows profiles.
- Workflow builder can add, edit, and remove toggle, direct-switch, cycle, and previous-profile workflows.
- Single-key trigger setup warns before saving.
- Config writes are validated before saving.
- Startup registration, setup reset/recovery, and process exclusions are available from tray/setup UI.
- Copy Diagnostics reports profile health, profile options, workflow readiness, and blocking reasons.
- Korean Hangul/native remains the first supported safe IME mode adapter.

## Acceptance Criteria

v0.2 is releasable when:

1. A clean user profile can create a non-Dutch/Korean workflow without editing JSON.
2. A user can configure a regular layout to regular layout toggle.
3. A user can configure a regular layout to Korean Hangul toggle.
4. A user can configure at least a three-profile cycle.
5. RightAlt/AltGr warnings are visible before saving that trigger.
6. Missing or ambiguous profiles are explained in diagnostics.
7. Existing v0.1 configs still load or migrate without behavior changes.
8. GitHub Actions passes.
9. Manual Windows validation is documented in `VALIDATION.md` and completed for the release build.
10. A portable Windows x64 ZIP is attached to the GitHub Release.

## Drift Rules For v0.2

- Release blocker: breaks setup, config generation, profile matching, or existing v0.1 runtime behavior.
- Same-stage improvement: improves setup/profile picking without expanding into a full settings editor.
- Future-stage item: full settings editor, installer, updater, per-app memory, extra IME mode adapters.
- Research item: TSF identity details, only until a concrete implementation choice is needed.
- Distraction: anything unrelated to creating reliable first multilingual configs.

Every PR must state which v0.2 implementation slice it advances.
